真实服务实验 (2026-05-22, 远程 GR00T via VS Code :8000 端口转发):
  - 端到端跑通：lerobot-eval -> vllm_groot -> 真实 GR00T(libero_sim) -> LIBERO；
    真实推理 ~1.2s/call，~43s/episode；reduced run (1 任务1 集) exit 0。
  - 两个必需修复（已落地）：
    (1) images 必须发**列表** [external_0, wrist]，不能发 dict（服务端直接喂 HF image processor）。
        config 新增 images_as_list=True；_build_request 改为发列表。
    (2) 序列化必须用 **OpenPI 的 msgpack-numpy 格式**（__ndarray__ 信封），与标准 msgpack-numpy 包
        的 nd/type/kind 格式不兼容。已**内置(vendor)**进 client.py，不依赖 openpi-client
        （后者 pin numpy<2，与 lerobot 冲突）。numpy 维持 2.2.6。
  - 默认 embodiment 改为 libero_sim（LIBERO 专用 projector id 2）。
  - 当前 pc_success=0 的真正原因（已查明）：
    (a) 服务端加载的是 **base** nvidia/GR00T-N1.7-3B；LIBERO 是 posttrain embodiment，
        需换成 **nvidia/GR00T-N1.7-LIBERO**（服务端：MODEL=nvidia/GR00T-N1.7-LIBERO 重启）。
        已通过 /v1/models 确认当前 root=nvidia/GR00T-N1.7-3B。
    (b) 之前 client 用了错误的 DROID eef_9d 约定。已改为 **LIBERO 原生 7 维约定**（权威来源
        Isaac-GR00T/gr00t/eval/sim/LIBERO/libero_env.py 与 examples/LIBERO/modality.json）：
        - modality/state/action key = x,y,z,roll,pitch,yaw,gripper（state.gripper 2 维）。
        - state 直接来自 lerobot LiberoProcessorStep 的 8 维 [eef_pos,axisangle,gripper_qpos]，
          图像已 180° 翻转（与 Isaac-GR00T 的 [::-1,::-1] 一致）→ flip_images_180=False。
        - images 必须发**列表** [agentview, eye_in_hand]。
        - 动作：Isaac-GR00T 把模型输出当作**delta**直接喂 env.step(relative)；gripper 做
          normalize([0,1]->[-1,1])->sign->invert。
        - vllm-omni 服务端 decode 会 **+raw_state**（已用 state.x=100 实测确认，返回≈100）→
          返回的是 current+delta（绝对）。故 client 用 decode_subtract_state=True 还原 delta，
          再走 LIBERO relative 控制；gripper 减去 state 后再 normalize/sign/invert。
      => client 侧已就绪：select_action→(1,7) 有限且∈[-1,1]。

  序列化关键修复：服务端用 openpi 的 msgpack-numpy 格式（__ndarray__ 信封），与标准 msgpack-numpy
  包不兼容；已**vendor**进 client.py（不引 openpi-client，避免 numpy<2 冲突）。
