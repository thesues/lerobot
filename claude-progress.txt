Task scope: 新增 `ws` 传输 backend —— controller 侧起一个专门的 ws_server_daemon 做数据中枢,
  媒体(JPEG 帧,BINARY)+ 控制/状态/动作(TEXT JSON)全经它中转;两端都拨出连它。
  解决「云 Pod ↔ 家用 Mac 两端都是对称 NAT」下 aiortc P2P 打不通、又不想上 LiveKit 的场景。
  aiortc(直连)/ livekit(SFU)两条老路径**完全保留、零改动**。

== 设计(分支 webrtc-ws-daemon)==
- 拓扑:mac_daemon(role=robot)──ws 推 state+frames / 收 actions──▶ ws_server_daemon
  (公网,ALB)◀──ws── controller(role=controller,同 Pod 本地)。两端都是 client 主动拨出 →
  落在「一端公网」可穿透情形,无需打洞/STUN/TURN。见 NAT_TRAVERSAL_NOTES.md。
- 复用现有 Transport 抽象:新增 WsRelayTransport(Transport),实现同一接口
  (open/channel/send_frame/set_frame_handler/connected/closed),CaptureAgent /
  _ProxyEndpoint / WebRTCProxyRobot 全不改(只认接口)。
- ws_server_daemon.py(新文件):近乎 dumb 转发器,按 (session,role) 配对。
  TEXT 给未到对端有界缓冲(maxlen=256,保住 hello + 早期 control);BINARY 只留最新一帧
  (丢陈旧,绝不堆积);对端断开给幸存者发 {"kind":"bye"}。只依赖 aiohttp(不解码帧,无需 cv2)。
- 连接握手:open() 直接发 {"kind":"hello"};收到对端 hello → connected。publisher 在
  connected 前不发 state/frames(避免 relay 端缓冲堆积)。fast-recycle 走 transport 的 bye→closed。
- 视频压缩:双 codec(两端 --ws-codec 必须一致),帧线格式 = 8字节大端 seq + payload:
  · jpeg(默认,帧内):cv2.imencode('.jpg',q=ws_jpeg_quality默认80);收端 imdecode。
    每帧独立可解→writer 只留最新未发帧(丢陈旧)。裸帧900KB→~30-80KB(~10-20Mbps)。
  · h264(帧间,PyAV/libav):av.CodecContext libx264(或 hwaccel h264_videotoolbox)zerolatency
    无 B 帧;publisher 对端到场后才编码→流首帧必为 IDR,新订阅端首包即可解(无需 keyframe 信令)。
    严格有序包流不能丢单帧→FIFO,积压超 _H264_MAX_BACKLOG=90 清空+强制 IDR 重同步。~1-5Mbps(3-10×省)。
    --ws-bitrate 码率,--ws-hwaccel 硬编码。色彩保真已独立验证(无 R/B 交换:红217/24/20…)。
  权衡:jpeg 帧内更费带宽,h264 帧间省但需有序流;带宽敏感优先 h264 或 aiortc/livekit,别随手降分辨率。
- 接线:make_transport 加 `ws` 分支(ws_url/ws_session/ws_token/ws_jpeg_quality);
  CaptureAgent / _ProxyEndpoint / run_daemon / WebRTCProxyRobot.connect 透传 ws 参数,
  复用 signaling_url(此时指向 ws daemon)/session_id/signaling_token;mac_daemon +
  cloud_teleop_so100 + config 的 --transport / transport_backend 增加 "ws" 选项。
